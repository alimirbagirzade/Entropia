About This Prototype

This prototype is a local browser-based interface study designed to demonstrate how a strategy development environment may be organized inside a single workspace. The purpose of the screen is not to execute real trading logic at this stage, but to make the relationship between strategy construction, backtest execution, result inspection, metric selection and future operational modules visually understandable.

The upper menu area works as a stable navigation shell. It remains visible while the lower workspace changes according to the selected menu item. This keeps the prototype simple and avoids the need for separate document files or separate browser pages. The current structure is intended to communicate the product logic, screen hierarchy, interaction model and functional grouping before any deeper implementation work begins.

The Mainboard area represents the central working surface. Strategy rows are added there, opened there, reviewed there and removed there. The backtest result area also appears inside this same workspace so that the user can immediately read the practical relationship between what was tested and what result was produced.

Reusable units are treated separately from result outputs. Strategy components, indicators, conditions and signal definitions may later become reusable structures. Backtest output, however, is treated as a result view rather than a reusable package.

Performance Metrics is intended to become the configuration and review area for measurement logic. It may contain result history, metric arrangement, visible metric selection, comparison preferences and other performance-related display choices.

Market Data is the registered source layer for OHLCV and execution-relevant market datasets. Research Data is a separate agent research layer for time-based supporting datasets such as funding, open interest, order book, liquidity and liquidation context.

Future development sections are displayed as visual placeholders. They are included to show where later operational modules may live, but they are intentionally inactive in this prototype stage.
